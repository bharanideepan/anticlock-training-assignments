# Data Model: Playwright E2E Test Suite

**Phase**: 1 — Design
**Date**: 2026-06-14
**Feature**: 006-playwright-e2e-tests

---

## Test Entities

These are not new data models — they describe how the test suite uses existing application entities for fixture creation and teardown.

---

### TestUser

Represents a pre-provisioned application user whose credentials are available via CI environment variables.

| Field | Type | Source | Notes |
|-------|------|--------|-------|
| email | string | `E2E_ADMIN_EMAIL` / `E2E_READONLY_EMAIL` env vars | Never hardcoded in test code |
| password | string | `E2E_ADMIN_PASSWORD` / `E2E_READONLY_PASSWORD` env vars | Never hardcoded in test code |
| role | RoleName enum | Determined by which account is used | |
| storageState | file path | `.auth/admin.json` / `.auth/readonly.json` | Generated by global setup; gitignored |

**Instances used by tests**:
- `adminUser` — `SYSTEM_ADMINISTRATOR` role; used for all positive CRUD scenarios
- `readonlyUser` — `READ_ONLY` role; used for RBAC denial tests

---

### TestCustomer (maps to `Customer` entity)

Created in test setup, deleted in teardown. Used for customer CRUD, search, filter, and navigation tests.

| Field | Type | Value Strategy | Validation Rule |
|-------|------|---------------|-----------------|
| companyName | string | `E2E-Customer-{workerIndex}-{timestamp}` | Required; unique (uniqueness key per spec Q2) |
| status | CustomerStatus | `PROSPECT` (default) | Enum: PROSPECT, ACTIVE, INACTIVE, ARCHIVED |
| industry | string? | `Technology` | Optional |
| ownerId | string | Admin user ID (resolved at setup) | Required |

**Uniqueness**: `companyName` must be unique. The timestamp + workerIndex suffix guarantees no collision across parallel workers or test reruns.

**Duplicate test data**: To test duplicate rejection, a second create attempt with the same `companyName` is made *within the same test* after the first record is confirmed created.

---

### TestContact (maps to `Contact` entity)

Created in test setup, deleted in teardown. Used for contact CRUD, form validation, and duplicate tests.

| Field | Type | Value Strategy | Validation Rule |
|-------|------|---------------|-----------------|
| firstName | string | `E2EFirst` | Required |
| lastName | string | `E2ELast-{workerIndex}-{timestamp}` | Required |
| email | string? | `e2e-{workerIndex}-{timestamp}@test.invalid` | Optional at DB level; validated as email format by form |
| customerId | string | ID of a TestCustomer created in the same test's setup | Required (FK to Customer) |

**Uniqueness**: Email is nullable in the database schema but the form validates the format. Duplicate email tests verify that the form or API rejects a second contact with the same email under the same customer.

**Note**: `.invalid` TLD ensures test email addresses can never be confused with real addresses and are clearly identifiable in logs.

---

### TestOpportunity (maps to `Opportunity` entity)

Created in test setup, deleted in teardown. Used for opportunity CRUD and pipeline tests.

| Field | Type | Value Strategy | Validation Rule |
|-------|------|---------------|-----------------|
| name | string | `E2E-Opp-{workerIndex}-{timestamp}` | Required |
| customerId | string | ID of a TestCustomer created in the same test's setup | Required |
| stageId | string | First available pipeline stage ID (resolved at setup) | Required |
| ownerId | string | Admin user ID (resolved at setup) | Required |

## Application Entities Referenced (Read-Only)

These entities are read by tests but not created or deleted by the test suite.

### PipelineStage

Tests that create Opportunities need an existing pipeline stage. The test setup resolves the first available stage via `GET /api/v1/pipeline/stages` and caches the ID for the test session.

### Role

RBAC tests assert that the `READ_ONLY` role cannot access create/edit/delete functionality. Role definitions are read from the authenticated user's JWT payload — not created by tests.

## State Lifecycle

```
Test Run Start
  └─ global-setup.ts
       ├─ Authenticate adminUser → .auth/admin.json
       └─ Authenticate readonlyUser → .auth/readonly.json

Per Test (create-from-scratch model)
  ├─ beforeAll / beforeEach
  │    └─ ApiHelper.createCustomer() → stores ID in test scope
  │    └─ ApiHelper.createContact()  → stores ID in test scope
  │    └─ ApiHelper.createOpportunity() → stores ID in test scope
  ├─ test body (assertions against UI)
  └─ afterAll / afterEach
       ├─ ApiHelper.deleteOpportunity(id)
       ├─ ApiHelper.deleteContact(id)
       └─ ApiHelper.deleteCustomer(id)

Test Run End
  └─ playwright-report/ and test-results/results.xml generated
```

## Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `BASE_URL` | No | Application base URL (default: `http://localhost:8080`) |
| `API_URL` | No | Backend API base URL (default: `http://localhost:3000`) |
| `E2E_ADMIN_EMAIL` | Yes (CI) | Admin test user email |
| `E2E_ADMIN_PASSWORD` | Yes (CI) | Admin test user password |
| `E2E_READONLY_EMAIL` | Yes (CI) | Read-only test user email |
| `E2E_READONLY_PASSWORD` | Yes (CI) | Read-only test user password |

All variables are read from process environment. **Credentials must never be hardcoded in test files.**

## Fixture Naming Convention

All test-created records use the prefix `E2E-` to allow easy identification and emergency cleanup:
- Customers: `E2E-Customer-{workerIndex}-{Date.now()}`
- Contacts: `E2EFirst E2ELast-{workerIndex}-{Date.now()}`
- Opportunities: `E2E-Opp-{workerIndex}-{Date.now()}`
- Emails: `e2e-{workerIndex}-{Date.now()}@test.invalid`

Emergency cleanup query (run manually if teardown fails):
```sql
DELETE FROM "Contact"  WHERE "firstName" = 'E2EFirst';
DELETE FROM "Opportunity" WHERE name LIKE 'E2E-Opp-%';
DELETE FROM "Customer" WHERE "companyName" LIKE 'E2E-Customer-%';
```
